#**“三核两翼”**的汇智AI产品体系：

* **应龙**：算力基座
* **驺吾**：数字基座
* **谛听**：智能体平台
* **精卫**：智能运维平台
* **白泽**：安全防护平台

三核负责生产力，双翼负责可用性与可控性。



## 一、总体架构

```text
                   ┌─────────────────────────────┐
                   │         统一智能体入口        │
                   │ 查询 / 档案 / 情报 / 案件 / 管控 │
                   └────────────┬────────────────┘
                                │
        ┌───────────────────────┼────────────────────────┐
        │                       │                        │
        ▼                       ▼                        ▼
┌────────────────┐   ┌────────────────────┐   ┌────────────────┐
│ 驺吾：数字基座   │   │ 谛听：智能体平台     │   │ 应龙：算力基座   │
│ 数据/标签/知识/  │◀▶│ Task IR / Plan GRAPH │◀▶│ OpenAI标准模型服务│
│ 样本四类工程     │   │ MCP / Skills / 记忆 │   │ 网关/调度/训练/治理│
└────────────────┘   └────────────────────┘   └────────────────┘
        ▲                       ▲                        ▲
        └───────────────统一执行与回写闭环───────────────┘

      横向产品：精卫（智能运维） / 白泽（安全防护）
```

这套架构的边界要明确：

* **应龙负责“能力供给”**
* **驺吾负责“资产沉淀”**
* **谛听负责“任务规划、执行、协作与进化”**
* **精卫负责“AI产品体系系统和网络的运维闭环”**
* **白泽负责“身份、权限、审计、风控与安全防护”**

核心思想不是“所有应用都变成聊天”，而是 **所有应用统一运行在智能体模式之上，但按场景采用不同复杂度的智能体形态**。

## 二、应龙：算力基座

### 2.1 产品定位

应龙最核心的目标只有一个：
**以统一协议，把模型能力、推理能力和训练能力稳定供给给上层。**

这里的“统一协议”是 **在私有化环境内，对外暴露 OpenAI 风格的标准接口**。这样应用、谛听、甚至第三方内部系统都可以用同一套 SDK、同一套调用方式对接。OpenAI 官方当前已将 `Responses API` 作为新项目推荐接口，并强调其支持多轮状态、多模态、工具调用与 remote MCP；因此，应龙为了生态良好，同样以 `/v1/responses` 为主接口，同时保留 `/v1/chat/completions` 的兼容面。

### 2.2 北向接口标准

应龙北向接口分两层。

**第一层：标准兼容层**

* `POST /v1/responses`：主接口
* `POST /v1/chat/completions`：兼容接口
* `POST /v1/embeddings`
* `GET /v1/models`
* `POST /v1/audio/transcriptions`
* `POST /v1/audio/speech`

这样设计的意义在于：
上层不关心底下是 vLLM、TensorRT-LLM、自研 ASR、OCR 还是多模态服务，只关心它是不是一个标准模型能力端点。

**第二层：内部扩展层**

所有不属于 OpenAI 标准接口的能力，不混进主协议，而统一进入内部扩展命名空间，例如：

* `/x/evals`
* `/x/trace`
* `/x/rerank`
* `/x/classify`
* `/x/predict`
* `/x/model-capabilities`
* `/x/scheduler`
* `/x/queue-status`
* `/x/eta-forecast`

这样既保证北向标准化，又保留业务场景所需扩展。

### 2.3 核心模块

应龙包含六个子模块。

#### 1）统一模型服务网关

负责：

* OpenAI 协议适配
* 模型路由
* 流式输出
* 统一鉴权
* 限流配额
* 请求追踪
* 能力发现
* 多租户隔离

这一层是应龙的“门面”，必须自研掌控，不依赖单一 serving 框架。

#### 2）推理服务池

负责：

* LLM / VLM / ASR / OCR / Embedding / Rerank / TTS 的具体 serving
* 多 runtime 插拔
* GPU / CPU / NPU 资源池隔离
* 热加载与灰度切换
* 多模型并存
* 统一健康检查

应龙的关键是 **网关统一、接口统一、调度统一**，而不是某一个开源组件。

#### 3）算力调度中心

应龙不只是 serving 平台，而是 **私有化场景下的算力规划平台**。

目前根据业务调研，固定三类队列：

* **在线队列**：低延迟、高优先级、为 Copilot/副驾/交互式请求保底
* **离线 VIP 队列**：高价值批任务、紧急专项任务、保障型 ETA
* **离线普通队列**：常规批处理、补算、重建索引、回算任务

调度中心应具备四个能力：

* 队列优先级与资源配额控制
* 基于任务画像的 placement 与并发控制
* 离线任务完成时间 ETA 预测
* 基于全局负载变化的 “what-if” 容量模拟

也就是说，应龙不仅调度“现在该跑谁”，还要回答“如果增加一批 OCR 回算任务、如果新增一组微调训练、如果提升 VIP 队列保障线，整个集群完成时间会怎么变化”。
这会直接给建设部门提供全局视野，而不是只看到局部 GPU 利用率。

#### 4）训练与适配平台

负责：

* LoRA / SFT 微调
* Embedding / Rerank 训练
* 小模型训练
* 蒸馏与压缩
* 多模型版本管理
* 奖励模型训练
* 偏好优化


#### 5）评测发布中心

负责：

* 基准评测
* 场景评测
* 回归测试
* 灰度发布
* 回滚
* 同任务排行榜

#### 6）模型治理中心

负责：

* 模型注册
* 版本治理
* 资源配额
* 成本统计
* 使用审计
* 发布审批
* 生命周期管理

### 2.4 明确边界

**应龙不负责：**

* Prompt 模板
* Few-shot 样例
* 业务 SOP
* 任务规划提示词
* 业务技能包
* 场景工作流定义

这些都应该进入谛听的 **Skills Registry / Workflow Registry / Memory Policy / Planner Policy**。
应龙主要做一层通用控制：

* 请求规范化
* 输出 schema 校验
* token 控制
* 安全基线
* trace 注入

## 三、驺吾：数字基座

驺吾按四类工程组织：

1. **数据工程**
2. **标签工程**
3. **知识工程**
4. **样本工程**

它的本质不是做“更多存储”，而是把现有的 Kafka / Flink / ETL / Hive / HBase / MPP / ES 体系升级成 **对象化、证据化、可学习化的数字世界底座**，成为沟通大数据到智能化原生应用的桥梁。

### 3.1 数据工程：把数据流加工成对象化事实

数据工程负责把现有数据链路从“表加工”升级为“事实生产线”。

它形成四层产物：

**原始层**

* 原始结构化数据
* 原始文本 （短文本、对话、群聊等）
* 图片、音频、视频、网页、Doc/PDF/PPT/excel 文件
* 原始采集元数据

**标准层**

* 统一数据契约
* 字段标准化
* 编码标准化
* 时间空间标准化
* 质量校验
* 主键与来源标识

**对象层**

把表和文档映射成公安对象：

* 人
* 车
* 号
* 设备
* 地点
* 账号
* 组织
* 案件
* 事件
* 线索
* 任务

**证据层**

把所有事实挂到证据链上：

* 来源系统
* 采集时间
* 加工任务
* 加工模型版本
* 命中规则/提示词
* 可回放引用

这层最终形成的核心资产不是单纯表，而是：

* 对象主索引（档案）
* 事件流 （时序）
* 关系边 （图谱）
* 证据文档
* 特征输入
* 向量化原料

### 3.2 标签工程：把对象语义层投影成查询友好界面

标签工程的目标不是替代对象语义层，而是把对象语义层物化成最适合当前用户使用习惯的查询界面。

标签库应被定义为：

> **对象语义层面向查询、筛选、统计、订阅和权限控制的高性能投影层**

标签工程包含：

* 标签定义中心
* 标签计算引擎 （四类标签关系定义）
* 标签存储与索引 （标签图）
* 标签运营
* 标签服务层
* 标签反馈闭环

标签工程的价值要重新定位：
**数字基座最重要的产品化界面之一。**
因为对很多GA、西元、ZZ业务来说，标签本身就是工作语言，是对象语义层面向一线使用者的高速视图。

### 3.3 知识工程：把事实和知识组织成可检索认知层

驺吾内部要区分两个概念：

* **对象语义层**：operational truth，强调对象、关系、事件、动作
* **知识工程**：cognitive layer，强调解释、规则、案例、文档、外部资料、检索组织

知识工程可以任务是数据工程结果的萃取和组织，包含五类资产：

#### 1）本体与语义模型

定义：

* 对象类型
* 关系类型
* 事件类型
* 证据类型
* 动作类型
* 约束与继承

#### 2）对象档案与关系图谱

形成：

* 对象档案
* 时间线
* 关系图
* 证据支撑图
* 案件/情报工作区知识视图

#### 3）文档知识库

沉淀：

* 线索材料
* 制度规范
* SOP
* 规则标准
* 历史A件/线索/未加工QB
* 外部公开资料
* 多模态解析文本

#### 4）向量库与检索层

采用双层检索：

* **关键词/过滤/聚合层**：ES
* **语义/跨模态 ANN 层**：Milvus

这样做的意义很直接：

* 过滤、权限、聚合继续复用现有检索体系
* 语义检索和跨模态检索走向量能力
* 文档、图片、音频片段、视频切片都可召回

#### 5）知识服务层

对谛听暴露：

* 事实检索
* 语义检索
* 跨模态检索
* 档案聚合
* 关系展开
* 时间线重建
* 证据引用

这一层优先以 MCP Server 方式暴露。MCP 本身就是面向 LLM 应用与外部数据源、工具之间的开放协议；现行规范使用 JSON-RPC，标准传输为 `stdio` 与 `Streamable HTTP`，HTTP 授权遵循 OAuth 2.1 约定。

### 3.4 样本工程：把反馈沉淀为学习资产

样本工程是整个体系的学习中心。

驺吾会固定五类样本：

* **监督样本**：微调、抽取、分类、匹配、排序
* **Prompt 样本**：few-shot、模板、拒答、证据引用
* **评测样本**：回归、安全红队、解释性验证、工具调用正确性
* **轨迹样本**：任务分解、工具选择、成功路径复用
* **偏好与反馈样本**：DPO / reward、纠错、误报修正、人工偏好

样本工程的输出同时服务三处：

* 给应龙：训练、微调、评测
* 给谛听：skills 调优、planner 训练/调优、tool routing 训练/调优
* 给驺吾：标签纠偏、知识补全

## 四、谛听：智能体平台

谛听是整套体系的中枢。
它不是聊天机器人平台，而是 **任务操作系统**。产品定义为：

> **统一承接自然语言规划、可视化编排和直接调用三种入口，并将其编译到同一 Task IR / Plan GRAPH，再由统一 Runtime 执行、审计、回写和进化的平台。**

### 4.1 三种构建形态

#### 形态 A：自然语言规划

用户一句话发起目标，平台通过多轮交互，完成：

* 对象识别
* 约束解析
* 计划拆解
* 工具编排
* 证据组织
* 结果生成

适合：

* 查证核查
* 线索研判
* 情报挖掘

#### 形态 B：拖拉拽可视化任务编排

用户通过算子拼装：

* 查询对象
* 展开关系
* 过滤时间窗
* 规则命中
* 语义检索
* 摘要生成
* 审批
* 回写
等
适合：

* 稳定 SOP
* 专题研判
* 多人协同流程
* 强审计场景

#### 形态 C：直接调用

不走复杂规划，直接触发某个 skill 或某个确定性流程：

* 档案聚合
* 标签解释
* 情报模板生成
* 固定核查动作

适合：

* 高频
* 高确定性
* 低延迟
* 风险边界清晰的场景

### 4.2 统一任务中间表示：Task IR / Plan GRAPH

这是谛听的核心技术资产。
自然语言、可视化拖拽、直接调用，最终都编译到同一套 Task IR / Plan GRAPH。

最小节点集合包括：

* Query
* Retrieve
* Expand
* Filter
* Join
* Summarize
* Generate
* Approve
* Writeback
* Notify

这样做的好处是：

1. 不同产品形态共用同一执行内核
2. 任务可审计、可回放、可比较
3. 自然语言生成的计划可以回落到可视化 GRAPH 编辑
4. 直接调用只是一个退化的单节点 GRAPH

谛听本质上不是“聊天 UI”，而是 **Task Compiler + Execution Runtime**。

### 4.3 四种运行层级

为了避免所有场景都被过度 Agent 化，谛听内建四级复杂度：

* **L0：直调式 Skill**
  一个请求，调用一个技能或确定性流程

* **L1：助手式 Agent**
  检索、RAG、结构化输出、简单工具调用

* **L2：工作流 Agent**
  显式 GRAPH、多步执行、条件分支、审批、回写

* **L3：规划式 Agent**
  动态任务拆解、动态选工具、并行子任务、多轮补查

原则很简单：
**默认选择最小足够智能级别。**

### 4.4 谛听的全链路观测能力

谛听内置一套 **Langfuse-like 的智能体观测面**。 将 tracing 定义为记录 LLM 应用请求全生命周期的能力，覆盖提示词、响应、检索、工具调用、时延、token、成本、元数据，并提供 prompt management、evaluation、experiments/datasets、自定义仪表板，而且支持自托管。Langfuse 还支持按 session 组织多轮交互，并强调 SDK 以异步队列方式上报 tracing 数据，以尽量降低对在线时延的影响。

因此需要采集：

* Trace / Span / Session
* Prompt 版本
* Skill 版本
* MCP 调用链
* 证据引用链
* Memory 命中链
* 模型调用链
* 审批动作链
* 成本、token、时延
* 在线 eval 与离线 replay

但要注意边界：
**谛听负责“智能体级观测”，AI体系产品的智能运维产品精卫负责“跨产品级运维观测”。**

### 4.5 谛听的记忆中心

记忆不能再做成简单的聊天历史，谛听把记忆设计成 **分层 + 图化 + 向量化** 的统一系统。整个memory 按层组织，让系统“在正确的时间记住正确的信息”；同时其 graph memory 会从每次 memory write 中抽取实体与关系，把 embeddings 写入向量库、把关系写入图后端，在检索时让语义命中结果与关系上下文同时返回。

谛听可以据此设计四层记忆：

* **任务记忆**：本次执行的临时上下文
* **会话记忆**：当前工作区或当前会话的连续状态
* **事件记忆**：围绕案件、情报、目标对象工作区的长期上下文
* **组织记忆**：组织级 SOP、历史经验、偏好和约束

并配套三类存储：

* 向量记忆：语义召回
* 图记忆：实体关系与事件关系
* 结构化记忆：状态、标签、偏好、配置

这样，谛听的记忆优势不只是“记得更多”，而是：

* 记得更分层
* 记得更可控
* 记得可追溯
* 能为多智能体协作共享上下文

### 4.6 谛听的智能体进化中心

谛听相比于其他智能体平台，如openclaw及其变种，manus等，核心差异化能力是“强化学习 + 无参数优化”构成的持续进化能力。这是因为toG的业务更聚焦，更具有领域学习潜力。

谛听的进化机制分成两条主线：

#### 1）无参数优化

由谛听平台内直接完成，重点优化：

* Prompt 模板选择
* Few-shot 动态装配
* 工具路由策略
* 任务模板召回
* 轨迹复用
* 检索策略与阈值
* 记忆写入/召回策略
* Reviewer / Guard 阈值

这条线不改模型参数，但会显著提升实际效果，且最适合私有化生产环境。

#### 2）参数化优化

谛听定义优化目标和奖励逻辑，应龙执行训练与发布，形成闭环。重点场景包括：

* Planner 策略优化
* Tool-use policy 优化
* 停止条件优化
* Reviewer 打分模型优化
* Reward model / DPO
* 离线 RL

注意：
谛听的**强化学习优化的是任务规划、工具选择、搜索深度、终止条件和质量判断，不是让模型自动生成高风险执法结论。**

因此，谛听设“进化中心”，负责：

* 轨迹采集
* 偏好建模
* 奖励定义
* 在线 A/B
* 离线 replay
* 无参数优化上线
* 参数训练任务提交给应龙
* 新版本回灌上线

### 4.7 智能体集群协作机制

谛听不只支持“单 Agent 串行调用和执行节点的简单多智能体协作”，而且支持 **智能体集群协作**。

采用“主控 + 专家群 + 守卫”的协作机制：

* **Commander**：接收目标，生成任务骨架
* **Planner**：生成/修正 Plan GRAPH
* **Retriever**：对接驺吾完成对象、标签、知识、向量检索
* **Analyst**：完成归纳、交叉验证、假设组织
* **Writer**：形成结构化输出与业务文书
* **Reviewer**：做结果复核、证据充分性检查
* **Guard**：校验权限、策略、风险动作
* **Executor**：负责回写、通知、任务流转

它们之间通过三种机制协作：

* **共享工作区黑板**：共享对象、证据、计划、草稿
* **消息邮箱**：子任务分发、状态回传
* **统一记忆引用**：按任务 / 会话 / 案件作用域读取上下文

这样平台的优势会非常明确：

* 单 Agent 适合轻任务
* 群体协作适合复杂案件和多模态研判
* 全部协作过程可回放、可审计、可比较

### 4.8 标准化复用与写作机制

这一部分做成谛听最重要的产品壁垒。

#### 第一层：工具标准化

统一采用 MCP 暴露数据、工具和资源。MCP 是面向 LLM 应用与外部数据源/工具对接的开放协议，适合作为谛听北向和驺吾南向之间的稳定接口；其标准传输为 JSON-RPC over `stdio` 或 `Streamable HTTP`，HTTP 授权可按 OAuth 2.1 落地。

#### 第二层：Skill 标准化

每个 Skill 不是一段 prompt，而是一个标准包，至少包含：

* `skill.yaml`：元数据
* `SKILL.md`：业务说明
* `input/output schema`
* `prompts/`
* `fewshots/`
* `tools/`
* `policies/`
* `evals/`
* `tests/`
* 版本号与发布记录

#### 第三层：Workflow 标准化

所有工作流统一编译为 Task IR / Plan GRAPH，并支持：

* YAML/DSL 编写
* 可视化拖拽
* 自然语言生成后再编辑

#### 第四层：编报写作标准化

对于情报编报、金融周报、线索总结、核查总结等写作类输出，统一采用：

* schema 化结构
* 证据强引用
* 事实/假设/分层
* 模板化文风
* Reviewer 审校
* 审批签发

支持模版的自定义和从历史资料学习，平台输出的不只是“写得像”，而是“写得规范、引用清楚、可审签”。

#### 第五层：Eval 与发布标准化

所有 Skill、Workflow、Prompt、Planner Policy 都必须具备：

* 测试样本
* 离线评测
* 回归门禁
* 灰度发布
* 回滚机制

这部分一旦建立起来，谛听就不仅是运行平台，更是 **组织级智能体资产工厂**。

## 五、精卫：智能运维平台

精卫是跨三核的横向产品，负责把“可运行”升级为“可稳定运行”。

它主要完成：

* 全链路拓扑
* 跨产品 SLO / SLA
* 故障告警
* 容量管理
* 资源画像
* 性能回归
* 自动诊断
* 回放与复盘
* 成本与算力利用分析

精卫的关键能力包括：

### 5.1 三类观测面统一

* **应龙观测**：模型网关、推理池、三队列调度、ETA 预测、GPU 利用率
* **驺吾观测**：数据流时延、标签回算、向量构建、索引新鲜度、知识服务可用性
* **谛听观测**：Task GRAPH、Agent trace、Skill 成功率、工具错误率、记忆命中率、审批耗时

### 5.2 全局容量与 ETA 视图

精卫要消费应龙的调度数据，形成建设部门最关心的能力：

* 当前集群总容量与缺口
* 在线保障线是否足够
* 离线 VIP 与普通队列积压
* 如果追加任务，ETA 如何变化
* 如果扩容/缩容，整体 SLA 如何变化

这使精卫不仅是运维平台，更是 **算力规划驾驶舱**。

### 5.3 智能诊断与自愈

精卫要支持：

* Trace 异常聚类
* 故障影响面定位
* 热点 Skill / Tool 漏斗分析
* 模型版本回退
* 调度策略回调
* 自动限流与熔断

## 六、白泽：安全防护平台

白泽是整个体系的“可信边界”。

它覆盖三核与精卫，负责：

* 身份与认证  （与管理中心系统）
* 权限与策略
* 内容安全
* 工具安全
* 审批与审计
* 高风险动作防护


### 6.1 身份与权限

采用统一的：

* IAM
* RBAC + ABAC
* 工作区隔离
* 对象级与字段级权限
* Skill / Workflow / MCP Server 级授权

对于 MCP，HTTP 传输场景下的授权可按 OAuth 2.1 体系落地；本地 `stdio` 场景可采用环境凭证或本地嵌入式凭证模式。MCP 官方文档也明确将 OAuth 2.1 作为受保护 MCP 服务的推荐约定。([模型上下文协议][5])

### 6.2 动作安全

白泽需要对以下动作实施门禁：

* 标签写回
* 对象合并/标记
* 任务下发
* 研判结果签发
* 对外通知
* 大范围检索
* 高风险工具调用

### 6.3 内容与模型安全

包括：

* Prompt 注入防护
* 工具越权防护
* 输出越权检测
* 敏感信息识别与脱敏
* 多模态内容风险检测
* 外部知识引入审计

### 6.4 审计与可追责

所有高风险链路必须记录：

* 用户
* Agent
* Skill
* Prompt 版本
* 模型版本
* MCP 工具
* 数据来源
* 审批链
* 写回结果

## 七、三核两翼的协同闭环

### 7.1 运行闭环

```text
统一智能体入口
   ↓
选择模式（直调 / 可视化流程 / 自然语言规划）
   ↓
谛听编译为 Task IR / Plan GRAPH
   ↓
谛听 Runtime 协同执行
   ├─ 调驺吾（对象/标签/知识/样本/记忆）
   ├─ 调应龙（OpenAI标准模型服务）
   ├─ 经白泽校验权限与风险
   └─ 由精卫监控链路与SLO
   ↓
结果回到工作区
   ↓
反馈沉淀到驺吾样本工程
```

### 7.2 进化闭环

```text
应用反馈 / 人工纠错 / Agent轨迹 / 在线评测
   ↓
谛听进化中心
   ├─ 无参数优化：模板、路由、记忆、检索、GRAPH策略
   └─ 参数优化：奖励建模、偏好建模、训练任务提交
   ↓
应龙训练/评测/发布
   ↓
新能力回灌谛听
```

### 7.3 运维与安全闭环

```text
运行数据 / 队列数据 / Trace / 审批 / 异常 / 安全事件
   ├─ 精卫：观测、诊断、容量、回放、自愈
   └─ 白泽：权限、审批、拦截、审计、追责
```

## 八、应用统一到智能体模式

统一到智能体模式，并不意味着都做成长链自治 Agent，而是 **所有应用共享谛听平台，但按场景选择最小足够复杂度**。

* **查询应用**：默认 L1 助手式 Agent
  保持原有查询体验，同时增加解释、补查、汇总

* **档案应用**：默认 L0/L1 混合
  档案聚合走直调，深入分析切到副驾

* **案件核查**：默认 L2 工作流 Agent
  可视化 SOP + 审批 + 结果回写

* **情报研判 / 案件侦办**：默认 L2/L3 混合
  固定流程可视化，复杂问题由规划式 Agent 补查

* **管控核查**：默认 L2 工作流 Agent
  规则清晰、审批清晰、回写清晰

完成智能体产品体系的构建之后，

> **所有业务应用都不再单独造平台，而是作为谛听之上的不同场景轻量化定制化运行，能够快速、灵活、精准的满足客户智能化业务需求。**
