## DeerFlow 存储与会话管理实现调研

### 1. 核心架构

DeerFlow基于**LangGraph + LangGraph Server**构建，其存储与会话管理设计如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                        Frontend (Next.js)                        │
└─────────────────────────┬───────────────────────────────────────┘
                          │ SSE / HTTP
┌─────────────────────────▼───────────────────────────────────────┐
│               LangGraph Server (port 2024)                       │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  Checkpointer (Async Provider)                              │ │
│  │  ├── memory   → InMemorySaver (进程内)                      │ │
│  │  ├── sqlite   → AsyncSqliteSaver (文件持久化)               │ │
│  │  └── postgres → AsyncPostgresSaver (多进程部署)             │ │
│  └─────────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  ThreadState (状态管理)                                      │ │
│  │  ├── messages, sandbox, thread_data                        │ │
│  │  ├── title, artifacts, todos                               │ │
│  │  └── viewed_images (带合并策略的reducer)                     │ │
│  └─────────────────────────────────────────────────────────────┘ │
└─────────────────────────┬───────────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────────┐
│               Gateway API (port 8001)                            │
│  ├── /memory     → 长期记忆存储                                  │
│  ├── /artifacts  → 工件文件存储                                  │
│  └── /uploads    → 用户上传文件存储                              │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 错误处理中间件 ([tool_error_handling_middleware.py](file:///Users/lzc/TNTprojectZ/deer-flow/backend/packages/harness/deerflow/agents/middlewares/tool_error_handling_middleware.py))

DeerFlow实现了专门的错误处理中间件：

```python
class ToolErrorHandlingMiddleware(AgentMiddleware[AgentState]):
    def wrap_tool_call(self, request, handler):
        try:
            return handler(request)
        except GraphBubbleUp:
            raise  # 保留LangGraph控制信号
        except Exception as exc:
            # 将异常转换为Error ToolMessage，让Agent可以继续执行
            return self._build_error_message(request, exc)
```

**特点**：
- 捕获工具执行异常
- 转换为结构化Error ToolMessage
- 记录详细日志 (tool name, id, exception)
- 保留LangGraph控制流信号 (interrupt/pause/resume)

### 7. 存储架构总结

| 数据类型 | 存储方式 | 持久化 |
|---------|---------|--------|
| **Agent状态** | LangGraph Checkpointer | SQLite/Postgres/InMemory |
| **消息历史** | LangGraph Checkpointer (ThreadState.messages) | 同上 |
| **长期记忆** | Gateway API → memory.json | JSON文件 |
| **上传文件** | Gateway API → uploads/ | 文件系统 |
| **生成工件** | Gateway API → artifacts/ | 文件系统 |
| **执行追踪** | LangSmith | 云服务 |

**DeerFlow当前未集成Langfuse**，而是使用LangSmith进行可观测性追踪。如果需要集成Langfuse，可以：

1. 参考 [tracing_config.py](file:///Users/lzc/TNTprojectZ/deer-flow/backend/packages/harness/deerflow/config/tracing_config.py) 的设计模式

## LangGraph Server 需要付费吗？需要联网吗？

LangGraph Server 是 完全免费且开源 的，基于 MIT 许可证。

### 关于付费
- 核心功能 ：LangGraph Server 本身是免费的，不需要任何订阅费用
- 依赖服务 ：如果使用外部 LLM 模型（如 OpenAI、Anthropic 等），则需要相应的 API 密钥和费用
- 部署选项 ：可以在本地机器、私有服务器或云服务上自由部署
### 关于联网
- 本地部署 ：LangGraph Server 可以在完全离线环境中运行
- 功能限制 ：
  - 离线环境下只能使用本地部署的模型（如 Llama 3 本地版）
  - 若需使用云服务 LLM 或外部工具（如 Web 搜索），则需要联网
- 数据隐私 ：本地部署时，所有会话数据都存储在本地，完全可控
### 技术实现
从 DeerFlow 项目的配置可以看出，LangGraph Server 支持：

- 本地文件存储（SQLite）
- 内存存储（开发/测试用）
- 数据库存储（PostgreSQL，多进程部署）
因此，LangGraph Server 是一个灵活的开源解决方案，可以根据需要选择离线或联网模式。

## 此项目的LangGraph Server是如何离线本地部署的？

LangGraph Server 在 DeerFlow 项目中可以通过以下方式实现离线本地部署：

### 1. 基本部署步骤

**1. 安装依赖**
```bash
# 在项目根目录
make install  # 安装所有依赖（前端 + 后端）
# 或仅安装后端
cd backend && make install
```

**2. 配置文件修改**
编辑 `config.yaml` 文件：
```yaml
# 模型配置 - 使用本地模型
models:
  - name: local-model
    type: openai  # 或其他支持的类型
    base_url: http://localhost:8080/v1  # 本地模型服务地址
    api_key: dummy  # 本地模型通常不需要真实API密钥

# 检查点配置 - 使用本地存储
checkpointer:
  type: sqlite
  connection_string: checkpoints.db  # 本地文件存储

# 沙箱配置 - 使用本地模式
sandbox:
  mode: local  # 本地执行模式
  local:
    enabled: true
```

**3. 启动服务**
```bash
# 开发模式（热重载）
make dev
# 生产模式
make start
```

### 2. 离线环境配置

**本地模型集成**
- 部署本地 LLM 服务（如 Ollama、vLLM 等）
- 配置 `base_url` 指向本地模型服务
- 确保模型支持所需功能（思考、视觉等）

**存储配置**
- **SQLite**：`connection_string: checkpoints.db`（文件存储）
- **内存**：`type: memory`（开发/测试用）
- **文件系统**：上传文件和生成工件存储在本地目录

**网络隔离**
- 关闭需要网络的工具（如 web 搜索、外部 API 等）
- 在 `extensions_config.json` 中禁用网络相关 MCP 服务

### 3. 技术实现细节

**LangGraph Server 配置**
- `langgraph.json` 中指定本地 checkpointer 提供器
- 通过 `make_checkpointer()` 函数创建本地存储实例

**离线工具支持**
- 沙箱工具（bash、文件操作等）完全离线
- 内置工具（任务管理、文件展示等）无需网络
- 本地技能库（`skills/` 目录）可离线使用

**数据持久化**
- 会话状态：SQLite 本地文件
- 长期记忆：`memory.json` 本地文件
- 上传/生成文件：本地文件系统

### 4. 验证离线部署

**启动验证**
```bash
# 检查服务状态
curl http://localhost:2026/api/langgraph/health
```

**功能测试**
- 创建本地会话，执行文件操作
- 使用本地模型生成内容
- 验证会话持久化（重启服务后检查）

**离线模式特点**
- 完全本地化，数据不出本地
- 无外部依赖，适合内网环境
- 性能取决于本地硬件和模型大小

通过以上配置，LangGraph Server 可以在完全离线的环境中运行，满足数据安全和网络隔离的需求。
